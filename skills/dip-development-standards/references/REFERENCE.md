Proper code management ensures traceability, collaboration, and reproducibility across teams and data products.Source ControlAll source code must be stored within Github Repos (or an approved equivalent under NZTA governance).No production code may reside solely in Databricks workspaces or on local machines. Workspace notebooks must be source-controlled via repo integration.Large artefacts (data files, models, binaries) must not be committed into repos; instead, store them in approved blob storage and reference them.Branching StrategyFollow a trunk-based branching model with a long-lived main (production) branch.Feature branches must be used for developing individual work items and merged into main via pull request (PR).The branch name must reflect the specific task or feature being worked on and be prefixed with a work item ID (e.g. feature/JIRA-5678-add-ccs-pipeline).RecommendationsStick to lowercase letters and using hyphens (-) to separate words, making the branch name easier to read. An exception for this is Jira work item IDs. For example, feature/ABC1234-add-ccs-cluster-configs, not Feature/ABC1234-AddCCSClusterConfigs.Be as specific as possible with your branch descriptions to avoid confusion. For example, instead of fix-error, use fix-ccs-quarantine-error.Branch TypesThe following are branch types that can be used to help differentiate development work and aid in workflow automation.Branch TypePurposeExampleFeatureNew features or enhancements to the codebase.feature/ABC1234-add-ccs-data-pipelineRefactoringRestructuring existing code without adding new features.refactor/DEF4321-move-ccs-to-sdp-pipelinesBug fixesFixing non-critical bugs or issues.bugfix/GHI5678-fix-duplicate-customersHotfixesCritical fixes that need to go directly to production.hotfix/GHI8765-fix-hr-data-duplicatesChores/dependenciesNon-functional tasks like updating dependencies or adding new documents.chore/JKL9012-add-ccs-pipeline-docsPerformance improvementTasks aimed at improving overall performance.perf/MNO1111-refine-job-cluster-configPull Requests and Code ReviewsAll merges to main must occur via pull requests.Peer review is required for all material code changes. Minor, low-risk updates (e.g., documentation edits, non-functional metadata updates, comment changes) may be merged without peer review where teams have agreed criteria. High-risk or production-impacting changes must always undergo review.PRs must include links to relevant Jira work items and documentation.Automated build/test pipelines must run on PRs; merges are blocked if tests fail.Commit HygieneClearly written commit messages are essential to keeping project history organized and intelligible. Adhering to recommended methods for crafting quality commits can greatly enhance teamwork and code quality, regardless of the scale of the project. Effective commit messages provide the following:Clarity - A clear description of what changes were made and why.Collaboration - Team members can understand easily the changes and the reasoning behind them.History - A detailed and understandable project history is created, making it easier to trace bugs, review code, and understand the evolution of the project.Revert Capability - The process of reverting specific changes if something goes wrong is greatly simplified.Commit Message StandardsCommit messages must be kept small and focused:One change per commit - Avoid bundling multiple changes into a single commit. For example, if you're fixing a bug and also refactoring code, these should be two separate commits.Frequent commits - Commit frequently to ensure that each change is well-documented and easy to understand.Commit messages must generally follow the Conventional Commits specification’s message structure (see examples below).Commit messages must be prefixed with one of the following types to help categorise the commit:TypeDescriptionExamplefeatIntroduces a new featurefeat(ccs): add access tablesfixFixes a bugfix(tm): resolve client-vehicle joins in silverchoreNon-functional changes (e.g., dependency updates)chore: update dependenciesdocsDocumentation changesdocs: update READMErefactorCode refactoring without functional changesrefactor: move copper merge to pythontestAdding or modifying teststest: add unit tests for copperstyleCode style changes (e.g., formatting)style: fix indentationperfPerformance improvementsperf: tune job cluster configbuildChanges to build process or toolsbuild: add prod targetPull requests must present a clean, reviewable set of commits. Developers may squash or rebase on their private branches, but must not rewrite shared history. Refs: ABC-1234Security and ComplianceNo secrets, keys, or credentials may be stored in code; use Azure Key Vault and environment variables instead.All Github repos must be configured with branch protection rules, mandatory PRs, and audit logging.Reproducibility and TraceabilityEach production release must trace back to a specific commit hash.All production jobs and pipelines must record their source commit in metadata.Code must be tagged and versioned for significant releases.Workspace IntegrationDatabricks notebooks must be linked to repos and follow the same branching/PR process as Python code.Ad-hoc notebooks may be used for exploration, but any developed production logic must be refactored into repos.CI/CD PracticesContinuous Integration and Continuous Deployment (CI/CD) processes are the backbone of modern engineering. CI/CD ensures that all code, pipelines, and infrastructure changes are automated, tested, secured, and governed, reducing manual intervention and deployment risk.Pipeline AutomationAll builds, tests, and deployments must be automated via Github Actions.Manual deployments or direct edits in production are prohibited.Pipelines must include build → test → security scan → package → deploy stages.Databricks Asset Bundles (DABs) must be used as the standard packaging and deployment format for Databricks assets (jobs, workflows, notebooks, clusters, policies).Environment PromotionChanges must progress through environments in sequence: Dev → Test → Prod.Promotion between environments must be automated, with approvals managed in Github.DABs must define environment-specific configurations (parameters, secrets, cluster sizes) to ensure consistency across environments.Security and DevSecOps IntegrationCI/CD pipelines must embed security controls:Static and dynamic security scanning (SAST, DAST).Dependency vulnerability scanning.Secret detection and validation.Pipelines must fail on security test failures — no overrides to push insecure code into production.Configuration and Secret ManagementAll environment secrets, connection strings, and credentials must be stored in Azure Key Vault, not in code or Github variables.Pipelines must retrieve secrets securely at runtime.DABs must be parameterised to use secrets injected from Key Vault.Infrastructure-as-Code (IaC)Platform infrastructure (such as clusters, networking, Unity Catalog policies) must be provisioned and managed exclusively with Terraform.CI/CD must validate Terraform templates with linting, policy checks, and plan/apply validation before deployment.Terraform state must be stored securely and version-controlled.Infrastructure changes must be peer-reviewed, traceable, and fully automated.Deployment GovernanceAll production deployments must require explicit approvals from authorised reviewers.Every production deployment must have an approved ServiceNow change request logged and closed out in accordance with NZTA process.Pipeline logs must capture who approved, when, and what was deployed.Rollback mechanisms must be built in — no irreversible deployments.Deployment metadata (commit hash, build number, DAB version, configuration) must be captured in Unity Catalog or equivalent.Observability and Feedback LoopsPipelines must emit telemetry on duration, failures, and deployment success.Alerts must trigger for failed builds, failed tests, or failed deployments.Post-deployment validations (smoke tests, data quality checks) must run automatically before the release is marked successful.Standardisation and ReuseCI/CD templates and YAML bundles must be standardised across teams.DAB-based deployment patterns must be used consistently to reduce drift across teams.Common pipeline checks (linting, testing, cost control, observability) must be embedded into shared templates.Code QualityHigh-quality code is essential for platform reliability, maintainability, and security. Code must be clear, consistent, and reviewed, ensuring that it can be easily understood, reused, and audited.Coding StandardsAll code must conform to agreed NZTA coding guidelines for each language.Python code must use type hints and docstrings for all functions and classes.Notebooks must separate SQL and Python cells clearly; reusable logic must be moved into libraries.Avoid “magic numbers” or hard-coded values — use configuration parameters.Linting and Static AnalysisLinting must run automatically in CI/CD (flake8, pylint, black for Python; lintr for R).Static analysis tools must enforce coding standards and flag security issues.Builds must fail if linting or static analysis reports critical issues.Logging and ObservabilityCode must use structured logging (JSON or key-value format) rather than print statements.Logs must include context (job ID, dataset, environment, user) to support debugging.Sensitive data must never be logged.Error Handling and ResilienceExceptions must be caught and handled gracefully with meaningful error messages.Failures must surface clearly in logs and monitoring dashboards.Retry policies and idempotent operations must be implemented for transient failures (e.g., API retries, cluster start delays).Code ReviewsAll code must undergo peer review before merge.Reviews must assess not only correctness but also readability, maintainability, security, and performance.Reviewers must check for appropriate tests and documentation updates.Security by DesignNo credentials, tokens, or secrets may appear in code.Code must be scanned for vulnerabilities (dependency checks, SAST).Unsafe functions and anti-patterns (e.g., eval) must be prohibited.Reusability and ModularityShared logic must be moved into libraries rather than copied across pipelines.Functions and modules must be small, single-purpose, and testable.Libraries must be versioned and documented for reuse.Performance and OptimisationSpark/Databricks code must follow optimisation best practices (partitioning, caching, avoiding UDF misuse).Large-scale transformations must be benchmarked before production release.Inefficient queries or transformations must be refactored during reviews.ReusabilityReusability reduces duplication, improves consistency, and accelerates delivery. All engineering teams must prioritise the creation of shared, modular, and discoverable assets across the Data Platform.Shared LibrariesCommon logic (e.g., data validation, logging, error handling, date functions) must be extracted into shared libraries, not duplicated across pipelines.Shared libraries may exist at team, domain, or organisation level depending on their scope of reuse:Team-level: utilities reused within a single team.Domain-level: components reused across a business domain (e.g., Transport, Enterprise, Regulatory).Organisation-level: components reused across multiple domains and maintained by the Data Platform teams.Organisation-level libraries require stricter governance, documentation, semantic versioning, and release processes.Libraries must be versioned, stored in Github Repos, and distributed via the platform’s package management approach (e.g. PyPI feed, R package repo).All shared libraries must include documentation and usage examples.Reusable ComponentsComponents such as ingestion frameworks, schema validation utilities, and monitoring hooks must be built for cross-product reuse.Teams must check for existing components before building new ones.Duplicate functionality must be consolidated into standard libraries at the appropriate tier (team, domain, or organisation).Templates and PatternsCI/CD pipelines, Terraform modules, and Databricks Asset Bundles (DABs) must use standard templates.Templates must include built-in testing, observability, and security checks.Patterns must be reviewed and published as platform standards to ensure consistency.Discoverability and DocumentationAll reusable assets must be documented and catalogued in SharePoint (governance/process) and Confluence (technical).Unity Catalog must link datasets to the reusable transformations or libraries that produce them.Documentation must include purpose, dependencies, versioning, and usage guidance.Cross-Language StandardsReusable assets are expected to be designed for reuse within their native language ecosystem (Python, R, SQL, Scala, Terraform, etc.).Teams are not required to implement libraries in multiple languages (e.g. both Python and R).Governance and MaintenanceReusable assets must have defined owners and maintainers.Deprecation policies must be applied to outdated libraries or patterns, with migration guidance provided.Data
